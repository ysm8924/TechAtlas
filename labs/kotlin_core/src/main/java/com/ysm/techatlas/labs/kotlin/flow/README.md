# Kotlin Flow 响应式数据流核心考点

Kotlin Flow 是基于 Kotlin 协程构建的响应式流（Reactive Stream），用于处理随时间推移发出的多个异步计算值。它是对标 RxJava 的现代化轻量级替代方案。

## 1. 核心概念：冷流 (Cold Flow) vs 热流 (Hot Flow)

### 1.1 冷流 (普通的 `Flow`)
*   **特性**：**“不订阅，不生产”**。只有当调用了终端操作符（如 `collect`, `first`, `toList`）时，Flow 内的代码才会开始执行。每次 `collect` 都会**重新执行**一遍完整的流构建流程。
*   **场景**：按需拉取数据，如数据库单次查询、网络单次请求。

### 1.2 热流 (`StateFlow` & `SharedFlow`)
*   **特性**：**“不管有没有人听，广播一直在播放”**。它独立于收集器存在，多个观察者（Collector）可以共享同一份数据和生产过程。
*   **场景**：全局状态管理、事件总线、UI 状态更新。

---

## 2. 状态与事件的双璧：StateFlow vs SharedFlow (面试最高频)

| 特性 | `StateFlow` (状态流) | `SharedFlow` (共享流) |
| :--- | :--- | :--- |
| **初始值** | **必须有**初始值 (必须传参) | **不需要**初始值 |
| **数据防抖** | 自动防抖 (Conflation)。连续赋同一个值，只会发射一次。 | 不防抖。赋同样的值也会全部发射出去。 |
| **重播机制** | 永远只重播**最新的一条**状态给新订阅者。 | 可配置 (`replay` 参数)，可以重播 0 条、1 条或 N 条。 |
| **Android 定位** | 完美替代 **LiveData**，用于维护 UI 状态 (如 Loading/Success)。 | 完美替代 **EventBus**，用于一次性事件 (如弹 Toast、导航跳转)。 |

---

## 3. 面试核心考点与坑点

### 考点一：Context 保护 (Context Preservation)
*   **坑点**：在 Flow 构建器 `{}` 内部，**绝对不能**通过 `withContext(Dispatchers.IO)` 来切换线程！这会抛出 `IllegalStateException`，因为 Flow 规定发射 (`emit`) 必须与收集 (`collect`) 处于同一上下文中。
*   **满分解法**：必须使用操作符 **`flowOn(Dispatchers.IO)`**。它会改变 `flowOn` **之上**的执行线程，而不影响其之下的代码。

### 考点二：异常处理 (`catch` 操作符)
*   **坑点**：`catch` 操作符是**透明的**，它只能捕获**它上游（在它之前）**发生的异常，对它下游（在它之后）发生的异常无能为力。
*   **实践**：通常把 `catch` 写在紧贴着 `collect` 的前一步。

### 考点三：背压 (Backpressure) 处理
当生产者的发射速度 > 消费者的处理速度时，就会产生背压。Flow 默认是**挂起**机制（消费者慢，生产者就会被挂起等待）。
*   **解法 1：`buffer()`**：加缓冲池，生产者继续发射存入池中，池满了才挂起。
*   **解法 2：`conflate()`**：丢弃中间值，只保留最新值（常用于视频帧处理）。
*   **解法 3：`collectLatest()`**：一旦有新数据发射过来，直接**取消**并重启当前正在缓慢执行的消费端代码（常用于搜索框防抖联想）。

## 4. 冷流 (Cold Flow) 进阶操作与原理

### 4.1 底层原理剖析
冷流的本质并不是什么神秘的对象池，它在底层就是一个**挂起函数 (Suspend Function)**的封装！
当你调用 `flow { emit(1) }` 时，你其实是创建了一个实现了 `Flow` 接口的对象，把代码块保存了起来。
当你调用 `collect { println(it) }` 时，实际上是触发了那个保存的代码块，并把你提供的 `{ println(it) }` 作为一个回调接口 (`FlowCollector`) 传进去执行而已。这就是为什么它**“不收集就不执行”**，以及为什么每次收集都会**重新跑一遍**的原因。

### 4.2 “平铺”与转换 (Flattening)
这是处理**多层异步嵌套**的终极武器。在 Android 中，最常见的坏代码就是 `collect` 里面套着 `collect`（产生地狱回调）。
*   **`flatMapConcat`**：串行。必须等上一个子流全部处理完，才处理下一个子流。（例如：先拉取文章列表，再依次拉取每个文章的详情）。
*   **`flatMapMerge`**：并发。不等待，来一个开一个并发去收集。
*   **`flatMapLatest`**：抢占。一旦有新的外层数据来到，立即**取消**旧的子流请求。极度适合【搜索框即时联想】功能！

## 5. Flow 核心操作符图鉴与面试必考点

操作符分为 **中间操作符**（返回新 Flow，不触发执行）和 **终端操作符**（挂起协程，触发执行）。

### 5.1 基础转换与过滤
*   **`map`**：最经典的一对一转换。（例如：JSON 字符串转 Object）。
*   **`filter`**：条件过滤。返回 true 才放行。
*   **`transform` (底层大杀器)**：`map` 和 `filter` 的底层都是基于它实现的。它最强大之处在于**它可以发射任意多次数据**，甚至不发射。

### 5.2 终端操作符 (`fold` vs `reduce` 经典考题)
*   这两者都是把流中所有的数据“揉”成一个最终结果（比如求和、拼接字符串）。
*   **`reduce` 坑点**：**必须有数据**。如果流是空的（比如 `emptyFlow().reduce(...)`），直接抛出 `UnsupportedOperationException`。
*   **`fold` 解法**：必须提供一个**初始值**。即便流是空的，它也会安全地返回初始值。在工程中更为推荐。

### 5.3 数据合并 (`zip` vs `combine` 经典考题)
这是大厂考察你是否真正写过复杂业务逻辑的分水岭。
*   **`zip` (严格拉链)**：
    *   **特性**：**严格的 1 对 1 等待**。Flow A 发了一个值，必须等到 Flow B 也发一个值，两者配对后才下发。
    *   **场景**：两份数据强绑定，比如拉取“商品列表”和对应的“价格列表”，必须一对一拼起来。
*   **`combine` (最新值组合)**：
    *   **特性**：**不等对方，各玩各的**。只要任何一个 Flow 发射了新值，它就会拉取另一个 Flow **最新**缓存的值，组合起来下发。
    *   **场景**：UI 状态刷新。比如“用户输入框 (Flow A)”和“复选框状态 (Flow B)”，只要任意一个变了，UI 就要根据最新的输入和勾选状态刷新按钮是否高亮。

### 5.4 时间维度操作符
*   **`debounce` (防抖)**：规定时间内没有新数据到来，才把最后一条数据放行。典型的“搜索框等待 500ms 后才发起请求”。
*   **`sample` (采样)**：固定每隔 X 毫秒，就去拿流里最新的数据放行。（类似滴答滴答的时钟，不管你发多快，我只按我的节奏取）。
