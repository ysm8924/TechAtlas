# Kotlin 与 Java 的类初始化与加载顺序深度剖析

类的加载与初始化顺序，是考察开发者对 JVM 底层运作机制和语言编译器行为掌握程度的经典试金石。

## 1. 核心铁律：从上到下 (Top-to-Bottom)
无论在 Kotlin 还是 Java 中，同一级别（静态级别 或 实例级别）的属性赋值和初始化代码块，**绝对遵循它们在源代码中出现的先后顺序（从上到下）执行**。

## 2. Kotlin 的初始化顺序全景图

当我们触发一个 Kotlin 类的加载与实例化时，顺序如下：

### 第一阶段：类加载期 (伴生对象 / 静态成员)
当类第一次被访问（调用伴生对象方法，或创建类实例）时触发，**仅执行一次**。
1. **Companion Object 属性与 `init` 块**：按代码书写的**先后顺序**从上到下依次执行。
   - *注意*：如果伴生对象在初始化某属性时（如 `val instance = Person()`），中途触发了类的实例化，那么此时会**暂停**伴生对象的后续初始化，先去走“第二阶段”，等实例创建完后，再回来继续往下走伴生对象剩下的 `init` 块！这就是你给出的代码中发生的神奇现象。

### 第二阶段：对象实例化期 (每次 new 都会走)
1. **主构造函数 (Primary Constructor)**：分配内存，主构造中的参数如果声明了 `val/var`，会在此刻分配。
2. **实例属性与实例 `init` 代码块**：这是 Kotlin 极易出错的考点！它们被合并在了一起，严格按照在类结构中**从上到下的书写顺序**交替执行。
3. **次构造函数 (Secondary Constructor)**：其方法体里的逻辑**最后执行**（必须先隐式或显式地调用主构造并跑完实例级别的初始化）。

---

## 3. Java 的初始化顺序全景图

Java 的规则与 Kotlin 极其相似，只是语法表现不同：

### 第一阶段：类加载期
1. **静态变量 (Static Variables)** 和 **静态代码块 (`static {}`)**：按源码先后顺序执行。

### 第二阶段：对象实例化期
1. **实例变量 (Instance Variables)** 和 **非静态代码块 (`{}`)**：按源码先后顺序执行。
2. **构造方法 (Constructor)**：最后执行。

## 4. 经典面试陷阱 (防坑指南)

1. **伴生对象/静态块的“提前插队”**
   - 就像你提供的例子：如果在伴生对象最上面写了 `val instance = Person()`，它会导致在类还没完全加载完毕时，就提前触发了对象实例化。此时实例被创建出来了，但伴生对象下面的 `init` 块都还没走！
2. **实例属性的“空指针谜团”**
   - 如果你在第一个 `init` 块中，去访问了写在它**下面**的属性，编译器可能会报错；如果绕过编译器强行访问，在 Java 中会拿到初始的零值（如 `null` 或 `0`），在 Kotlin 中则可能引发不可预知的行为。因此，**尽量将所有的属性声明放在 `init` 块的前面！**
