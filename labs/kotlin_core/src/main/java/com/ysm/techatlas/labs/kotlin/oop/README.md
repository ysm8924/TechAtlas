# 重载 (Overload) 与 重写 (Override) 核心考点剖析

在面向对象编程中，“重写”与“重载”是最基础也是最容易在高级面试中被结合起来设坑的知识点。理解它们的本质，核心在于理解**分派（Dispatch）**的时机。

## 1. 核心概念与底层差异

| 特性 | 重载 (Overloading) | 重写 (Overriding) |
| :--- | :--- | :--- |
| **发生范围** | 同一个类（或作用域）内 | 父子类（继承关系）之间 |
| **条件限制** | **同名不同参**（参数类型、个数不同） | **同名同参**（方法签名完全一致） |
| **决议时机** | **编译期决议 (Static Dispatch)** | **运行期决议 (Dynamic Dispatch)** |
| **底层指令** | 编译器在编译时根据传入的**声明类型**直接硬编码生成对应的签名（如 `invokevirtual` 带有确切的参数类型） | 运行时 JVM 根据对象的**实际堆类型**在虚方法表（vtable）中动态查找 |

## 2. 常见考点与陷阱

### 陷阱一：参数的静态决议（重载的坑）
当调用一个被重载的方法时，编译器到底选哪一个？
**答案**：编译器只看变量的**声明类型**，完全不关心它运行时到底指向什么对象。

### 陷阱二：泛型擦除导致的重载失败 (Kotlin 专属考点)
在 Java 中，不能同时拥有 `fun process(list: List<String>)` 和 `fun process(list: List<Int>)`，因为编译后泛型会被擦除为 `List`，导致签名冲突。
**Kotlin 解法**：利用 `@JvmName` 注解，在底层字节码层面强行给其中一个方法改名，从而绕过编译器的重载冲突检查。

### 陷阱三：属性的重写 (Kotlin 专属特性)
Kotlin 中不仅方法可以被重写，**属性也可以被重写**。
一个 `val`（只有 getter）可以被子类重写为 `var`（getter + setter），但这会引发一些关于底层字段（Backing Field）的有趣现象。

## 3. 面试终极必杀题：Double Dispatch（双重分派）
如果把“方法的重写（动态分派，看实际类型）”和“参数的重载（静态分派，看声明类型）”结合在一起，会发生什么？
**法则**：
1. 先看是哪个对象调用的方法 -> **动态决议（看 new 的真实对象）**。
2. 再看方法里传入了什么参数 -> **静态决议（看参数的声明类型）**。
