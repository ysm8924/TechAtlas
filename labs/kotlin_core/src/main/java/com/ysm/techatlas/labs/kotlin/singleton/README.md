# 单例模式 (Singleton Pattern) 深度实战与底层机制

单例模式是开发中最基础的设计模式，但在面试中却能牵扯出并发编程、类加载机制、甚至底层硬件指令重排的深度问题。

## 1. 实际使用场景
* **重量级对象**：如 `RoomDatabase` 数据库实例、`Retrofit` / `OkHttpClient` 网络请求客户端。这类对象创建极其耗时且消耗资源，全局只需一份。
* **全局状态/配置**：如 `AppConfig` 统一配置管理器、`UserManager` 用户登录状态管家。
* **无状态工具类**：如统一下发线程池的 `ThreadPoolManager`。

## 2. 核心考点与面试陷阱

### 考点一：双重检查锁 (DCL) 为什么需要 `volatile`？
* **Java 陷阱**：在 Java 中实现 DCL 时，如果不加 `volatile`，可能会拿到**半初始化**的对象。
* **底层原理 (指令重排)**：`new Instance()` 在底层分为三步：1.分配内存空间；2.初始化对象；3.将指针指向分配的内存。JVM 为了优化，可能会把步骤 2 和 3 重排（1 -> 3 -> 2）。如果线程 A 执行了 1 和 3，此时线程 B 进来，发现指针不为空，直接拿去用，就会引发 Crash！`volatile` 的核心作用之一就是**禁止指令重排**。

### 考点二：静态内部类单例的原理
* **为什么它既能懒加载，又天生线程安全？**
* **底层原理**：JVM 规范保证了在类加载（ClassLoader）的 `<clinit>()` 初始化阶段，是**线程互斥（自带锁）**的。当外部类被加载时，静态内部类并不会被加载。只有第一次调用 `getInstance()` 时，才会触发内部类的加载和静态变量的初始化。完美实现了懒加载和绝对的线程安全，且没有任何 `synchronized` 的性能损耗。（《Effective Java》极力推荐）。

### 考点三：枚举单例的绝对防御
* **陷阱**：常规单例可以通过**反射**强行调用私有构造，或者通过**反序列化**生成新对象，从而破坏单例。
* **防御**：枚举（Enum）单例在 JVM 层面被特殊照顾。Java 源码明确规定反射无法创建枚举实例；序列化机制也对枚举做了特殊处理，保证全局唯一。

### 考点四：Kotlin 的 `object` 有没有缺陷？
* **缺陷：无法传参！** Kotlin 的 `object` 声明不能有构造函数。如果在 Android 中，你的单例需要传入一个 `Context`（比如 Room 数据库），`object` 就无能为力了。
* **解决方案**：必须使用带有参数的 **DCL (伴生对象 + `@Volatile` + `synchronized`)**。

## 3. Java 与 Kotlin 对应关系映射

| 模式 | Java 实现 | Kotlin 实现 | 特点 |
| :--- | :--- | :--- | :--- |
| **饿汉式** | `public static final` 变量直接 new | **`object`** 关键字 | 类加载时初始化，写法最简单 |
| **懒汉 DCL** | `volatile` + `synchronized` | **`by lazy`** (默认模式) | 延迟加载，应对高并发 |
| **带参单例** | DCL (传入参数) | **`companion object` DCL** | 解决 object 无法传参的痛点 |
| **静态内部类** | 嵌套 `private static class` | 无直接等价物（通常用 lazy 代替）| 性能最优的无参懒汉式 |
